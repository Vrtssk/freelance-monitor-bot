# Filtering pipeline

Нормализованные посты из скраперов проходят через гибридный фильтр
`filters/pipeline.HybridFilter`, который комбинирует:

1. **Ключевой предфильтр** (`filters/keywords.match_keywords`) — быстрый, бесплатный,
   синхронный. Для каждой темы проверяется вхождение любого ключевого слова из
   `config/topics.TOPICS`. Пост, не совпавший ни с одной темой пользователя,
   отбрасывается до обращения к LLM.
2. **LLM-классификация** (`filters/llm.LLMClassifier`) — уточняющая, платная/лимитная.
   Запускается только для кандидатов, прошедших предфильтр.

## Протокол общения с LLM

Модели свободного уровня OpenRouter часто нестабильны (reasoning-модели
возвращают ответ в поле `reasoning`, а не `content`; формат JSON бывает
«broken»). Поэтому классификатор использует **простой текстовый формат** вместо
JSON. Системный промпт требует ровно одну строку:

```
RELEVANT: YES|NO; TOPICS: <список через запятую из допустимых категорий, или пусто>
```

Пример: `RELEVANT: YES; TOPICS: chatbots, parsing`.

`LLMClassifier._parse_response` читает только часть **после `TOPICS:`**, чтобы
категории, упомянутые в рассуждении (reasoning trace) модели, не давали ложных
совпадений. Темы фильтруются до множества `allowed_topics` пользователя.

Текст берётся из `msg.content`, а при пустом `content` — из `msg.reasoning`
(fallback для reasoning-моделей). Таймаут запроса — 30 c.

## Вакансии / найм (исключение)

Пользователю нужны **только фриланс-заказы** — разовые проекты либо
долгосрочное сотрудничество с исполнителем. Объявления о найме сотрудника на
работу (вакансии, собеседования, «по результатам собеседования», полная
занятость, оклад, «в штат», «требуется программист на работу») считаются
**нерелевантными** и жёстко отсекаются.

Детектор `filters/vacancy.is_vacancy(post)` — текстовая эвристика по
сильным маркерам найма (regex, case-insensitive, по `title + description +
category`). Он срабатывает **до** ключевого фильтра и LLM: вакансия не
попадает в `HybridFilter.filter_posts`, не уведомляется и помечается
`is_vacancy=True` в `seen_posts`. Дополнительно LLM-промпт (`filters/llm.py`)
явно просит отвечать `RELEVANT: NO` на вакансии — вторая линия защиты.

В Топ-5 (`db/repository.get_top_relevant`) вакансии исключены условием
`is_vacancy = False`, поэтому даже уже сохранённые вакансии (backfill при
`init_db`) выпадают из выдачи.

## Off-topic guard (узкие темы пользователя)

Темы пользователя узкие: веб-фронтенд, вёрстка сайтов, парсинг/сбор данных,
скрипты/автоматизация, чат-боты/Telegram-боты. Ключевой предфильтр
намеренно **широкий** (чтобы не пропустить кандидатов), поэтому слово
«верстка» ловит и «Верстку книжную» в мобильных приложениях. Чтобы такие
ложные срабатывания не попадали в Топ-5, после проверки вакансий стоит
детерминированный guard `filters/off_topic.is_off_topic(post)`:

- срабатывает на явные **внешние** домены — нативная разработка мобильных
  (iOS/Android) и десктоп-приложений (`ios`, `android`, `swift`, `kotlin`,
  `flutter`, `xcode`, «мобильное приложение» и т.п.);
- но **не** отсекает, если в посте есть in-scope сигнал одной из тем
  пользователя (например, «нужен telegram-бот внутри приложения» → тема
  chatbots). Признак «верстка» в guard НЕ считается in-scope, т.к. в контексте
  приложений это раскладка экрана, а не вёрстка сайтов.

Guard — вторая (после LLM-промпта) линия защиты от нетематических IT-заказов.

## Резервная логика (fallback)

`HybridFilter` гарантирует доставку уведомлений даже при недоступности LLM:

- **LLM выключен** (`LLM_ENABLED=false`) или без ключа → используются совпадения
  по ключевым словам (`match_reason="keywords"`).
- **Ошибка LLM** (rate limit, таймаут, сетевой сбой — `reason` начинается с
  `llm_error`) → пост отправляется по ключевым словам (`match_reason="keywords
  (llm unavailable)"`).
- **LLM отверг** (`RELEVANT: NO`) → пост отбрасывается.
- **LLM принял, но темы не извлеклись** при наличии `allowed_topics` → пост
  отбрасывается (нельзя уведомлять «без темы»).

## Провайдер LLM (Groq)

Классификатор провайдер-агностичен: использует OpenAI-совместимый клиент
(`openai.AsyncOpenAI`) с `LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL` из `.env`.
Текущий провайдер — **Groq** (`https://api.groq.com/openai/v1`, модель
`llama-3.1-8b-instant`), у которого значительно выше бесплатные лимиты, чем у
OpenRouter (раньше использовался `poolside/laguna-xs-2.1:free`, упирался в
50 запросов/день). При исчерпании лимита Groq конвейер корректно переходит на
ключевые слова (fallback), уведомления продолжают доставляться.

# Фильтрация и ранжирование

## Канонические темы

| Ключ | Название |
|---|---|
| `frontend` | Front-end |
| `markup` | Вёрстка сайтов |
| `parsing` | Парсинг и сбор данных |
| `scripts` | Скрипты и автоматизация |
| `chatbots` | Чат-боты |

Списки ключевых слов находятся в `config/topics.py`. В коде следует использовать
ключ `scripts`, а не `automation`.

## Пайплайн

`HybridFilter.filter_posts()` обрабатывает каждый пост последовательно:

1. `filters.vacancy.is_vacancy` жёстко исключает найм, штат и вакансии.
2. `filters.off_topic.is_off_topic` исключает явные внешние домены, например
   native mobile/desktop, если нет сильного сигнала нужной темы.
3. `filters.keywords.match_keywords` ищет ключевые слова только выбранных тем.
4. Для кандидатов запускается LLM, если он включён и настроен.
5. При отключённом LLM используются keyword matches.
6. При timeout/rate limit/ошибке LLM также применяется keyword fallback.
7. Явный ответ LLM `NO` исключает пост.

Такой порядок экономит LLM-запросы и сохраняет уведомления при недоступности
провайдера.

## LLM-протокол

Используется `openai.AsyncOpenAI` с настраиваемыми base URL и model. Текущие
значения по умолчанию ориентированы на Groq.

Модель должна вернуть одну строку:

```text
RELEVANT: YES|NO; TOPICS: frontend, scripts
```

Парсер принимает только допустимые темы пользователя. Ответ читается из
`content`, а для некоторых reasoning-моделей предусмотрен fallback на
`reasoning`.

## Вакансии и off-topic

Детерминированные guards работают до LLM. Это вторая линия защиты от ложных
совпадений вроде мобильной «вёрстки» или объявления о найме frontend-разработчика.
Вакансии сохраняются с `is_vacancy=True`, отображаются на общей веб-доске с
маркером и исключаются из top-выборок.

## Relevance score

`utils.relevance.compute_relevance()` возвращает число `[0, 1]`:

| Фактор | Вес | Чем лучше |
|---|---:|---|
| количество откликов | `0.55` | меньше конкуренция |
| сложность | `0.20` | проще проект |
| свежесть | `0.18` | новее публикация |
| цена | `0.07` | выше бюджет |

Сложность `1..5` оценивается эвристикой по описанию. Свежесть использует
`published_at`, а при его отсутствии — `seen_at`. Доллары приблизительно
переводятся по фиксированному коэффициенту 90; договорной бюджет не превращается
в число.

## Две top-выдачи

- Telegram `/top`: до 5 заказов, чьи сохранённые темы пересекаются с темами
  пользователя.
- Web `/top`: по умолчанию 10 лучших не-вакансий из максимум 500 кандидатов;
  выдача не персонализирована.

Обе выдачи пересчитывают recency в момент запроса.

## Ограничения

- LLM-классификация последовательна и может повторяться для разных пользователей.
- `matched_topics` хранится на глобальном посте, а не отдельно для пользователя.
- Keyword fallback повышает надёжность доставки, но менее точен, чем успешный LLM.
- Формула relevance эвристическая и не является вероятностью получения заказа.

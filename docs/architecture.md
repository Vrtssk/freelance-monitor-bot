# Архитектура Freelance Monitor

Этот документ описывает фактически реализованную архитектуру. Инструкции по
запуску находятся в [operations.md](operations.md), фильтрация и рейтинг — в
[filtering-and-ranking.md](filtering-and-ranking.md), сайт и HTTP API — в
[web-and-api.md](web-and-api.md).

## Процессы

Приложение запускается тремя Docker-сервисами:

```text
                   PostgreSQL
                  /          \
                 /            \
 Telegram Bot + Scheduler     FastAPI + Web Board
 aiogram, APScheduler         Jinja2, HTMX, Alpine
         |                           |
         +---- scrapers/filter ------+
```

- `bot` владеет экземпляром Telegram Bot, запускает APScheduler и отправляет
  уведомления.
- `api` отдаёт HTML-доску и JSON API. Это отдельный процесс, в нём нет
  экземпляра Telegram Bot и планировщик не запускается.
- `db` хранит пользователей, настройки, объявления и историю циклов.

## Поток мониторинга

`MonitorService.run_cycle()` выполняет один цикл:

1. Получает активных пользователей и их включённые источники.
2. Строит объединение источников: биржа парсится, если она нужна хотя бы одному
   активному пользователю.
3. Запускает соответствующие скраперы и пропускает глобально виденные посты.
4. Для каждого пользователя оставляет его источники и темы.
5. Применяет vacancy/off-topic guards, ключевые слова и LLM-классификацию.
6. В процессе `bot` отправляет совпадения в Telegram.
7. Сохраняет остальные новые посты как просмотренные и записывает `ScrapeRun`.

Одновременно выполняется не более одного цикла: APScheduler использует
`max_instances=1`, а сервис дополнительно защищён флагом `_running`.

## Компоненты

| Каталог | Назначение |
|---|---|
| `bot/` | aiogram handlers, клавиатуры и запуск polling |
| `api/` | FastAPI routes и подготовка Jinja-контекста |
| `templates/` | HTML-страницы и общий partial карточек |
| `config/` | настройки, темы и реестр источников |
| `db/` | SQLAlchemy-модели, сессии и repository-функции |
| `scrapers/` | FL.ru, Freelance.ru, Weblancer и Kwork |
| `filters/` | guards, ключевой и LLM-фильтры |
| `scheduler/` | цикл мониторинга и APScheduler |
| `utils/` | Telegram-форматирование и relevance score |
| `tests/` | pytest-тесты модулей и UI-рендеринга |

## Модель данных

### `users`

Telegram-пользователь, username, активность и флаг паузы мониторинга.

### `user_topics`

Выбранные пользователем темы. Пара `(user_id, topic_key)` уникальна.

### `user_disabled_sources`

Хранятся только отключённые пользователем источники. Если записей нет, включены
все биржи. Такой подход автоматически включает новые источники по умолчанию.

### `seen_posts`

Нормализованные объявления. Дедупликация глобальная по
`(source, external_id)`. Помимо текста хранятся темы, бюджет, дата публикации,
отклики, оценка сложности, числовая цена и признак вакансии.

### `scrape_runs`

История запусков скраперов: источник, время начала/окончания, найденные и новые
посты, ошибка. Веб-интерфейс использует последнее `finished_at` для live-таймера.

## Важное ограничение multi-user

Настройки тем и источников являются per-user, но `seen_posts` и поле `notified`
глобальны. Отдельной таблицы доставок `user_post_deliveries` пока нет. Поэтому
первый обработавший пост пользователь может повлиять на обработку того же поста
для следующих пользователей. Для надёжной многопользовательской эксплуатации
нужна отдельная per-user модель доставки.

## Инициализация схемы

При старте вызывается `Base.metadata.create_all()`. Для нескольких добавленных
позже колонок выполняются идемпотентные PostgreSQL `ALTER TABLE ... IF NOT
EXISTS` и backfill вакансий. Alembic и версионированных миграций сейчас нет.

## Ручные циклы

- Кнопка бота и `/check` запускают цикл в процессе `bot`, поэтому уведомления
  могут быть доставлены.
- `POST /scrape/run` запускает цикл в процессе `api`, где Telegram Bot
  отсутствует. Endpoint предназначен для локального управления/диагностики и
  способен сохранить посты просмотренными без Telegram-доставки.

Последний пункт особенно важен при эксплуатации: публичный API нельзя открывать
без внешней авторизации.
